Ingestion LayerProtocol-Specific, Shared Patterns

Ingestion Layer

Protocol-specific services; shared architectural conventions.

01

Dual-channel ingestion

gRPC streams and RPC polling operate in parallel per service.

02

Protocol-specific parsing

Transaction parsing and account decoding scoped per protocol domain.

03

Write-on-difference

Unchanged state suppressed to reduce load without sacrificing freshness.

04

Isolated async queues

Per data category — slow or failing writes never block others.

05

Dynamic account discovery

Monitored account sets evolve over time without service restarts.

06

Env-driven configuration

Centralised per-service — no bespoke one-off configuration patterns.

Query Layer

The frontend consumes parameterised SQL functions — never raw tables.

A stable application contract

  • Consistent query interfaces across pages & widgets
  • Flexible time windows and re-bucketing on demand
  • Clean separation of ingestion, transformation, presentation

Analytics composed close to data

  • Price-impact & liquidity traversal (AMM-specific)
  • Risk-sensitivity calculations in view functions
  • Protocol-specific AMM analytics, not duplicated in app code

Application Layer

Two coordinated FastAPI services — independently operable.

Resilience & RecoveryBounded Recovery & Escalation

Resilience & Recovery

Short-lived recovery inside the service; persistent failure escalates.

01

Internal reconnect

For transient gRPC and DB issues — bounded retry within the service.

02

Automatic thread restart

For failed subscription workers — isolated restart without full service teardown.

03

Platform restart

Via liveness failure or self-exit watchdog — escalation when internal recovery is exhausted.

Services do not retry indefinitely. Persistent failure escalates predictably instead of silently degrading.

  • Health endpoints on each service
  • Queue health & worker liveness checks
  • gRPC stream health monitoring
  • Backfill workflows for historical gaps

Design PrinciplesHow the Platform Is Engineered

Design Principles

Choices that make the system easier to operate, extend, and reason about.

01

Schema as documentation

Heavily annotated SQL definitions act as authoritative reference.

02

Single source of truth

Discriminators, config, schemas, and views each have one defined home.

03

Queue isolation

Slow or failing writes in one category do not block others.

04

Write-on-difference

Unchanged state is suppressed without sacrificing data freshness.

05

Bounded recovery

Services do not retry indefinitely; failure escalates predictably.

06

Platform-agnostic containers

Deployment works across environments with env-level changes only.
